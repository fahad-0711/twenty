## 2. TWENTY'S INTERNAL ARCHITECTURE — CRITICAL KNOWLEDGE

### 2.1 The Dual PostgreSQL Schema System

> ⚠️ THIS IS THE MOST IMPORTANT THING TO UNDERSTAND BEFORE WRITING ANY CODE.

Twenty uses TWO separate PostgreSQL schemas in the same database instance:

```
PostgreSQL (calllive_crm database)
│
├── schema: "core"                    ← Twenty's engine tables
│   ├── workspace                     ← Workspace config
│   ├── "user"                        ← User accounts
│   ├── object_metadata               ← Object definitions (the schema registry)
│   ├── field_metadata                ← Field definitions
│   └── relation_metadata             ← Relation definitions
│
└── schema: "workspace_{uuid}"        ← ALL workspace data lives here
    ├── company                       ← auto-generated by metadata engine
    ├── person                        ← auto-generated by metadata engine
    ├── opportunity                   ← auto-generated by metadata engine
    ├── [your custom objects]         ← if using metadata API
    └── _any_core_extension_tables_   ← where we put VoiceJourney etc.
```

**Implication for CallLive extensions**:
- Our new tables (VoiceJourney, WebhookEvent, RealEstateProject, etc.) go into the **workspace schema** — NOT the core schema.
- We CANNOT use standard TypeORM `migration:generate` for this — see Section 3.

### 2.3 The Migration System — CRITICAL UPDATE

> ⚠️ **DO NOT USE LEGACY MIGRATIONS.** Twenty has deprecated the TypeORM migration approach.

Recent commit: `chore(server): rename TypeORM migrations dir to legacy-typeorm-migrations-do-not-add`

This means: the `packages/twenty-server/src/database/migrations/` folder is **DEPRECATED**. Do not add files there.

**The correct approach for CallLive extensions** is:
1. For metadata objects (RealEstateProject as a custom object): Use Twenty's Metadata API mutations — no migration needed
2. For performance-critical custom tables (VoiceJourney, WebhookEvent): Use a **separate migration runner** — see `.claude/skills/typeorm-migrations/SKILL.md`

### 2.4 Two Separate GraphQL APIs

| API | Endpoint | Purpose | When to Use |
|-----|----------|---------|-------------|
| **Core API** | `/api` or `/graphql` | Workspace data (Companies, People, custom objects) | Querying/mutating CRM data |
| **Metadata API** | `/metadata` | Schema configuration (add fields, objects) | Adding custom fields to existing Twenty objects |

```graphql
# Core API — query workspace data
POST /api
{ "query": "{ companies { edges { node { id name } } } }" }

# Metadata API — configure schema
POST /metadata
{ "query": "mutation { createOneObject(input: {...}) { id } }" }
```

Our custom TypeORM entities are created in the **workspace schema**, using a dedicated `calllive_` prefix:

```sql
-- In schema workspace_{uuid}

calllive_webhook_event     ← WebhookEvent entity
calllive_real_estate_project
calllive_project_unit
calllive_site_visit
```

This prefix prevents collisions with Twenty's auto-generated tables and makes our extensions instantly identifiable.

Every CallLive.ai call maps to a VoiceJourney record. Here is the complete state machine:

```
                    ┌─────────────────┐
                    │   CALL ARRIVES   │
                    │ (webhook fires)  │
                    └────────┬────────┘
                             │ call.initiated
                             ▼
                    ┌─────────────────┐
         ┌──────────│    INITIATED    │──────────┐
         │          └────────┬────────┘          │
         │                   │ call.in_progress   │ call.failed
         │                   ▼                   │
         │          ┌─────────────────┐          │
         │          │   IN_PROGRESS   │          │
         │          └────────┬────────┘          │
         │                   │ call.completed     │
         │                   ▼                   ▼
         │          ┌─────────────────┐  ┌─────────────────┐
         │          │   COMPLETED     │  │     FAILED      │
         │          └────────┬────────┘  └─────────────────┘
         │                   │
         │                   │ transcript.ready (async, may be delayed)
         │                   ▼
         │          ┌─────────────────────────────┐
         │          │   COMPLETED + transcript    │
         │          │   transcript_raw populated  │
         │          └────────────────┬────────────┘
         │                           │
         │                           │ sentiment.analyzed (async)
         │                           ▼
         │          ┌─────────────────────────────────┐
         │          │   COMPLETED + ai_analysis       │
         │          │   sentiment_score               │
```

## 5. WEBHOOK INGESTION DATA FLOW

```
CalLive.ai Platform
      │
      │ POST /api/webhooks/calllive/events
      │ Headers: X-CallLive-Signature, X-CallLive-Event-ID
      │ Body: JSON payload
      ▼
┌─────────────────────────────────────────────────────────────┐
│  CallLiveWebhookController                                  │
│                                                             │
│  Step 1: Extract raw body (MUST use rawBody: true in app)  │
│  Step 2: CallLiveWebhookValidator.verifyHmac()             │
│          → HMAC-SHA256(raw_body, CALLLIVE_WEBHOOK_SECRET)  │
│          → Compare to X-CallLive-Signature header           │
│          → On fail: throw UnauthorizedException (HTTP 401) │
│                                                             │
│  Step 3: Zod.parse(body) → validate payload shape          │
│          → On fail: throw BadRequestException (HTTP 400)   │
│                                                             │
│  Step 4: await webhookEventRepo.upsert({ event_id })       │
│          → Store raw payload in calllive_webhook_event     │
│          → Use upsert (idempotent — same event_id = skip)  │
│                                                             │
│  Step 5: await queue.add('process-calllive-event',         │
│            { event_id, event_type }, { jobId: event_id })  │
│          → BullMQ deduplicates by jobId automatically      │
│                                                             │
│  Step 6: return { received: true }  [< 1 second total]    │
└─────────────────────────────────────────────────────────────┘
      │
      │ (async, separate BullMQ worker process)
      ▼
┌─────────────────────────────────────────────────────────────┐
│  CallLiveWebhookProcessor                                   │
│                                                             │
│  1. Fetch WebhookEvent by event_id                          │
│  2. Route to handler by event_type:                         │
│     • call.initiated     → handleCallInitiated()           │
│     • call.completed     → handleCallCompleted()           │
│     • call.failed        → handleCallFailed()              │
│     • call.no_answer     → handleCallNoAnswer()            │
│     • transcript.ready   → handleTranscriptReady()         │
│     • sentiment.analyzed → handleSentimentAnalyzed()       │
│  3. Each handler: upsert VoiceJourney by call_id           │
│  4. Contact matching: find Person by phone, or create new  │
│  5. Update WebhookEvent.processing_status = PROCESSED      │
│  6. If sentiment.analyzed: call AutoTaskCreatorService     │
└─────────────────────────────────────────────────────────────┘
```

## 8. DECISION TREE — WHERE TO PUT NEW DATA

When adding a new field or object, use this decision tree:

```
Is it high-volume data? (>10k records/year)
├── YES ──▶ Use TypeORM Entity in workspace schema (calllive_ prefix)
│           See .claude/skills/typeorm-migrations/SKILL.md
│
└── NO ──▶ Is it a new property of an EXISTING Twenty object (Company/Person/Opportunity)?
           ├── YES ──▶ Use Twenty's Metadata API
           │           (no migration, just GraphQL mutation to /metadata)
           │
           └── NO ──▶ Is it a new domain object with its own record page in Twenty UI?
                      ├── YES ──▶ Use Twenty's Apps Framework
                      │           npx create-twenty-app
                      │           Use defineObject() pattern
                      │
                      └── NO ──▶ Use TypeORM Entity in workspace schema
```
