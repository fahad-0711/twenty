## Question 1: Person Module Structure (end-to-end)

### Files Examined
- `packages/twenty-server/src/modules/person/` — only contains `standard-objects/person.workspace-entity.ts`

### Key Finding: Twenty Does NOT Use Traditional NestJS Modules for Domain Objects

Twenty's `person/` module is **not** structured like a classic NestJS module (with separate `.module.ts`, `.resolver.ts`, `.service.ts`, `.entity.ts` files). Instead, it uses a **metadata-driven architecture**:

```
packages/twenty-server/src/modules/person/
└── standard-objects/
    └── person.workspace-entity.ts    ← The ONLY file
```

#### How It Works

1. **Entity Definition** (`person.workspace-entity.ts`):
   - Extends `BaseWorkspaceEntity` (from `src/engine/twenty-orm/base.workspace-entity.ts`)
   - Defines fields as TypeScript properties with types from `twenty-shared/types` (e.g., `FullNameMetadata`, `PhonesMetadata`, `EmailsMetadata`)
   - Defines relations using `EntityRelation<T>` type (e.g., `company: EntityRelation<CompanyWorkspaceEntity>`)
   - Exports `SEARCH_FIELDS_FOR_PERSON` for full-text search configuration

2. **No Resolver or Service Files**:
   - There is no `person.resolver.ts` or `person.service.ts`
   - GraphQL queries/mutations for Person are **auto-generated at runtime** by Twenty's metadata engine
   - The engine reads `ObjectMetadata` records from the `core` schema → builds GraphQL types via `workspace-schema-builder` → creates FindMany/FindOne/Create/Update/Delete resolvers dynamically

3. **The Connection Chain**:
   ```
   PersonWorkspaceEntity (type definition)
       ↓ registered via Twenty ORM module
   ObjectMetadata (stored in core.objectMetadata table)
       ↓ read by workspace-schema-builder
   GraphQL Schema (auto-generated at runtime)
       ↓ resolvers use workspace-datasource.service
   SQL queries against workspace_{uuid} schema
   ```

4. **BaseWorkspaceEntity** provides `id`, `createdAt`, `updatedAt`, `deletedAt` fields automatically.

### Implication for CallLive
The `person/` module is NOT a template for creating custom TypeORM modules with resolvers and services. It's a workspace entity definition. For CallLive's custom modules (VoiceJourney, WebhookEvent), we need a **different pattern** — creating standalone NestJS modules with explicit resolvers and services, registered in `ModulesModule`.

### Answer: Can We Extend the Existing Webhook System?

**No.** Twenty's existing webhook infrastructure serves two unrelated purposes:
1. **Outgoing webhooks** (engine/metadata-modules/webhook/) — fire HTTP calls when CRM records change. This is the opposite of what we need.
2. **SES messaging webhooks** (modules/messaging-webhooks/) — receive AWS SES email notifications. Highly AWS-specific with SNS signature verification.

**We must build a new CallLive webhook module from scratch**, but the `messaging-webhooks` module is an excellent structural template:
- It uses `@Controller()` with `@Post()` route decorators
- It accesses `request.rawBody` for signature verification (critical for HMAC)
- It uses `@UseGuards(PublicEndpointGuard, NoPermissionGuard)` to bypass JWT auth
- It uses a custom exception filter pattern
- It's registered in `CoreEngineModule`, not `ModulesModule`

### Usage Pattern (from existing code)
```typescript
// Injecting a queue
@InjectMessageQueue(MessageQueue.webhookQueue)
private readonly webhookQueueService: MessageQueueService,

// Adding a job
await this.webhookQueueService.add<MyJobData>(
  MyJobClass.name,
  { eventId: '...', eventType: '...' },
);

// Processing a job
@Processor(MessageQueue.webhookQueue)

```

### Implication for CallLive
We should **NOT** import BullMQ directly. Instead, we should:
1. Add a new `callliveWebhookQueue` entry to the `MessageQueue` enum
2. Use Twenty's `@InjectMessageQueue`, `@Processor`, and `@Process` decorators
3. Follow the exact pattern used by `call-webhook.job.ts`

## Question 4: Where to Register New NestJS Modules

### Search Command
```bash
grep -r "PersonModule\|CompanyModule" packages/twenty-server/src --include="*.ts" -l
```

### Result: **No matches found**

There is no `PersonModule` or `CompanyModule` — because Person and Company are **workspace entities**, not NestJS modules. They don't have dedicated modules; they're auto-handled by Twenty's metadata engine.

### The Correct Registration Point

The module hierarchy is:

```
AppModule (src/app.module.ts)
├── CoreEngineModule (src/engine/core-modules/core-engine.module.ts)
│   ├── MessagingWebhooksModule    ← Incoming webhooks registered HERE
│   ├── WebhookModule              ← Outgoing webhooks
│   ├── MessageQueueModule         ← Queue system
│   └── ... (40+ core modules)
│
├── ModulesModule (src/modules/modules.module.ts)
│   ├── MessagingModule            ← Business logic modules registered HERE
│   ├── CalendarModule
│   ├── WorkflowModule
│   ├── ConnectedAccountModule
│   ├── OnboardingInviteSuggestionsModule
│   └── WorkspaceMemberModule
│
├── CoreGraphQLApiModule           ← Workspace data GraphQL
├── MetadataGraphQLApiModule       ← Schema config GraphQL
└── RestApiModule                  ← REST endpoints
```

### Where to Register CallLive Modules

**`src/modules/modules.module.ts`** — this is where business logic modules go. We add our new modules to the `imports` array:

```typescript
// src/modules/modules.module.ts
@Module({
  imports: [
    MessagingModule,
    CalendarModule,
    ConnectedAccountModule,
    OnboardingInviteSuggestionsModule,
    WorkflowModule,
    WorkspaceMemberModule,
    // === CALLLIVE EXTENSIONS ===
    CallLiveWebhookModule,    // Phase 3.1
    VoiceJourneyModule,       // Phase 3.3
    RealEstateProjectModule,  // Phase 3.4
    ProjectUnitModule,        // Phase 3.4
    SiteVisitModule,          // Phase 3.4
  ],
})
export class ModulesModule {}
```

> **Note:** The `MessagingWebhooksModule` (SES webhooks) is registered in `CoreEngineModule`, not `ModulesModule`. For CallLive's webhook controller, `ModulesModule` is appropriate since it's an extension, not a core engine component.

## Question 5: Metadata API vs TypeORM Entity for RealEstateProject

### Files Examined
- `engine/metadata-modules/object-metadata/object-metadata.entity.ts` — schema registry entity (174 lines)
- `engine/metadata-modules/object-metadata/object-metadata.resolver.ts` — CRUD resolver (266 lines)
- `engine/metadata-modules/object-metadata/object-metadata.service.ts` — service (36K)

### How the Metadata API Works

Twenty's Metadata API lets you create **custom objects** at runtime via GraphQL mutations:

```graphql
mutation {
  createOneObject(input: {
    object: {
      nameSingular: "realEstateProject"
      namePlural: "realEstateProjects"
      labelSingular: "Real Estate Project"
      labelPlural: "Real Estate Projects"
    }
  }) { id }
}
```

This creates a row in `core.objectMetadata`, which the `workspace-schema-builder` reads to dynamically generate GraphQL types, resolvers, and database tables in the workspace schema — all without writing any code.

### Trade-off Analysis

**Using the Metadata API** would give us instant GraphQL CRUD, automatic UI rendering in Twenty's record pages, built-in search/filter, and zero custom code — but we lose control over JSONB column types (like `transcript_raw`), custom enums (like `CallStatus`), complex indexing strategies, and the ability to run performant bulk queries on high-volume voice data (potentially millions of records).

**Using TypeORM entities** requires writing NestJS modules, resolvers, services, and migrations manually, but gives us full control over schema design, complex column types (JSONB, enums, composite indexes), and performance-tuned queries — which is essential for VoiceJourney (high-volume, JSONB transcripts, multi-column search) and WebhookEvent (write-heavy, idempotency constraints).

### Recommendation

| Entity | Approach | Why |
|--------|----------|-----|
| **VoiceJourney** | TypeORM entity | High-volume, JSONB transcript, custom enums, performance-critical queries |
| **WebhookEvent** | TypeORM entity | Write-heavy, idempotency constraint, raw JSONB payload storage |
| **RealEstateProject** | TypeORM entity | Complex relations (units, site visits), custom enums, kanban views |
| **ProjectUnit** | TypeORM entity | ManyToOne to Project, custom status enum, booking relations |
| **SiteVisit** | TypeORM entity | Multi-entity relations (Person, Project, Unit, VoiceJourney) |
| **Company extensions** | Metadata API | Adding simple fields (gst_number, rera_number) to existing objects |
| **Person extensions** | Metadata API | Adding simple fields (pan_number, role, source_channel) to existing objects |

## Summary: Key Decisions for Phase 2+

1. **Do NOT copy `person/` module structure** for new entities. Person uses the metadata-driven workspace entity pattern. Our custom modules need explicit NestJS modules with resolvers, services, and entities.

2. **The `messaging-webhooks` module is our structural template** for the CallLive webhook controller — it shows how to handle raw body, bypass auth, and register in the module tree.

3. **BullMQ is available** (`5.78.0`) behind Twenty's `MessageQueue` abstraction. Add a new queue name to the enum, use `@InjectMessageQueue` and `@Processor` decorators.

4. **Register new modules in `src/modules/modules.module.ts`** — this is the business logic module aggregator imported by `AppModule`.

5. **Use TypeORM entities for all 5 new CallLive tables**, use Metadata API only for extending existing Company/Person objects with simple fields.

6. **Must add a `calllive_` table prefix** per ARCHITECTURE.md to avoid collisions with Twenty's auto-generated tables.

7. **All custom tables go in the workspace schema** (`workspace_{uuid}`), not the `core` schema.
